# System Optimizer — Multi-Agent Pipeline Performance Engineering Skill

## Purpose

Provides a systematic framework for profiling, diagnosing, and optimizing the performance of the four-tier trading research pipeline. Covers context window budget management, agent coordination efficiency, parallel execution tuning, token cost reduction, and latency optimization. Used when the pipeline is running slowly, consuming excessive tokens, or producing suboptimal results due to coordination overhead.

## Stakeholder

Used by the **Project Lead** during pipeline execution to monitor and tune system performance. Referenced by **Orchestrators** when managing context handoff between scouts and verifiers. May be invoked by any agent that detects performance degradation or context budget exhaustion.

## Instructions

### 1. Performance Profiling Framework

#### 1a. Agent-Level Profiling

For each agent in the pipeline, track these metrics:

| Metric | Measurement | Target | Alert Threshold |
|--------|------------|--------|-----------------|
| **Token input** | Total tokens consumed by agent prompt + context | Varies by tier | > 50% of context window |
| **Token output** | Total tokens generated by agent | < 30% of context window | > 40% of context window |
| **Execution steps** | Number of tool calls made | < 15 per agent | > 25 per agent |
| **Source fetch latency** | Time spent on WebSearch/WebFetch | < 30s per fetch | > 60s per fetch |
| **File I/O operations** | Number of Read/Write operations | < 10 per agent | > 20 per agent |

#### 1b. Pipeline-Level Profiling

| Metric | Measurement | Target |
|--------|------------|--------|
| **Total scouts spawned** | Count including respawns | 8-10 (8 base + 0-2 respawns) |
| **Confirmation Gate pass rate** | % of scouts passing first attempt | > 75% |
| **Deduplication ratio** | Strategies removed / total ingested | 10-25% |
| **Verifier consensus** | % of strategies where score variance < 15 points | > 80% |
| **Pipeline wall-clock time** | Total execution time | Context-dependent |

### 2. Context Window Budget Management

#### Token Budget Allocation

The pipeline should allocate context window budget as follows:

| Component | % of Context | Purpose |
|-----------|-------------|---------|
| **System prompt + agent instructions** | 15-20% | Agent definition, rubric, schema |
| **Input data (scout outputs, enriched data)** | 25-35% | Strategies being processed |
| **Shared context (rules, IV data, calendar)** | 10-15% | Reference data loaded by all agents |
| **Tool call results** | 15-25% | WebSearch, WebFetch, Read results |
| **Output generation** | 15-20% | Agent's own output |
| **Buffer** | 5-10% | Safety margin for unexpected expansion |

#### Context Compression Strategies

When context usage exceeds 70% (as flagged by `context-window-monitor.sh`):

**Strategy 1: Selective context loading**
- Load only the strategies relevant to the current task (e.g., for a Weekly Verifier, don't load Monthly/Quarterly strategies)
- Load only the relevant sections of `rules/OptionsTrading.md` (e.g., lot sizes for the target instrument only)

**Strategy 2: Progressive summarization**
- Summarize completed scout outputs into key fields only: strategy_name, structure, underlying, expiry_category, edge_thesis
- Full details can be re-loaded from file when needed for deep analysis

**Strategy 3: Reference-based loading**
- Instead of embedding full shared_context.json, embed only the current IV regime and next expiry date
- Use file paths as references and load on-demand

**Strategy 4: Output streaming**
- Write output incrementally rather than building the full output in memory
- After writing each strategy's verification, clear the working memory for that strategy

### 3. Agent Coordination Efficiency

#### Parallel Execution Optimization

The pipeline has natural parallelism points:

| Stage | Parallel Opportunities | Dependencies |
|-------|----------------------|--------------|
| **Scout spawning** | All 4 scouts per bias can run in parallel | All depend on shared_context.json (read-only) |
| **Cross-bias scouting** | Bullish and Bearish ScoutLeaders are independent | Both depend on Lead's source list |
| **Verifier spawning** | All 3 Verifiers per bias can run in parallel | All depend on enriched output (read-only) |
| **Cross-bias orchestration** | Bullish and Bearish Orchestrators are independent | Both depend on rules and shared_context |

**Anti-patterns to avoid:**
- Sequential scout spawning when parallel is possible
- Loading all scout outputs into Orchestrator context when only one bias is being processed
- Re-fetching shared_context.json for every agent when it hasn't changed
- Running Verifiers sequentially when they have no interdependencies

#### Context Handoff Optimization

When passing context between tiers:

1. **Lead → ScoutLeader:** Pass only bias, source list, IV regime, expiry filter. NOT the full README or rules file (ScoutLeader can read these itself)
2. **ScoutLeader → Scout:** Pass only domain assignment, bias, output path, relevant source URLs. NOT the full source list for all domains
3. **Orchestrator → Verifier:** Pass only the enriched strategies for the target expiry category. NOT all enriched strategies across all categories
4. **Verifier → Lead:** Pass only the verification summary table and scores. Lead reads full details from file only for top candidates

### 4. Token Cost Reduction Strategies

#### High-Token Operations to Optimize

| Operation | Token Cost | Optimization |
|-----------|-----------|-------------|
| **WebFetch on long articles** | 2000-8000 tokens per page | Use WebSearch first to confirm relevance; only WebFetch the most promising 2-3 URLs |
| **Loading full rules/OptionsTrading.md** | ~1500 tokens | Load only relevant sections per agent role |
| **Scout output files** | ~2000 tokens each × 8 files | Orchestrator loads only its bias's 4 files, not all 8 |
| **Verifier rubric template** | ~800 tokens | Already embedded in agent file; don't re-load from README |
| **Repeated shared_context reads** | ~500 tokens per read | Read once, pass as parameter to subagents |

#### Model Selection (if configurable)

For token-intensive but lower-complexity tasks:
- **Scout research:** Standard model (needs web search capability)
- **Orchestrator dedup:** Can use faster model for similarity comparison
- **Verifier analysis:** Standard model (needs nuanced judgment)
- **Lead synthesis:** Standard model (needs strategic judgment)

### 5. Latency Reduction Techniques

#### Pre-warming

Before spawning scouts, the Lead should:
1. Pre-fetch the IV regime and store in shared_context.json
2. Pre-validate the source list (check which URLs are accessible)
3. Pre-load the expiry calendar

This prevents each scout from independently fetching the same data.

#### Result Caching

Within a single pipeline run:
- Cache WebSearch results for common queries (e.g., "India VIX current level")
- If multiple scouts search for the same instrument's data, the Lead should pre-fetch and store in shared_context.json
- Verifiers for the same bias can share pre-fetched SEBI compliance data

#### Early Termination

- If a scout returns 5 high-quality strategies on the first attempt, don't respawn even if some edge cases weren't covered
- If deduplication removes >50% of strategies, flag to the Lead but don't trigger additional scout spawns unless strategies drop below minimum
- If all Verifiers score a strategy as REJECT, skip the correlation check for that strategy

### 6. Quality vs. Speed Tradeoffs

| Configuration | Speed-Optimized | Quality-Optimized |
|--------------|----------------|-------------------|
| Scout respawn max | 1 | 3 |
| WebFetch per scout | 2-3 URLs | 5-8 URLs |
| Historical scenarios per strategy | 1 | 3+ |
| Dedup similarity threshold | 12/15 | 10/15 |
| Verifier debate depth | Summary only | Full advocate/adversary simulation |
| Cross-market data search | Skip | Include with adaptation notes |

The default pipeline runs in **quality-optimized** mode. Use `--mode` flag adjustments for speed optimization when time-constrained.

### 7. Performance Monitoring During Execution

The Lead should log performance data in `run_state.json`:

```json
{
  "performance": {
    "scouts_spawned": 8,
    "scouts_respawned": 1,
    "confirmation_gate_attempts": 1,
    "strategies_ingested": 39,
    "strategies_after_dedup": 32,
    "strategies_verified": 32,
    "strategies_in_final": 18,
    "warnings_generated": 4,
    "estimated_total_tokens": "context-dependent"
  }
}
```

### 8. Common Performance Issues and Fixes

| Issue | Symptom | Fix |
|-------|---------|-----|
| Context overflow mid-run | Agent stops producing coherent output | Trigger summarization; reduce context passed to next tier |
| Scout produces fewer than 5 strategies | Respawn consumes additional tokens | Pre-validate source URLs in Lead's source research step |
| Duplicate strategies inflate pipeline | Orchestrator processes redundant strategies | Move lightweight dedup check to ScoutLeader tier |
| Verifier scores wildly inconsistent | Lead must spawn tie-breaking Verifier | Ensure all Verifiers receive identical rubric and context |
| WebFetch returns SPA shells | Scout wastes tokens on empty content | Use `web-fetch-analyzer` skill proactively; maintain a blocklist |
| Rate-limited by search providers | Scout stalls on web research | Space requests; use cached results; fall back to alternative sources |
